# 离线资料备份基础

## 用途

介绍离线系统中资料备份的基础原则，包括多副本、异介质、校验、冷备份和恢复演练，减少数据损坏或设备故障带来的知识丢失。

## 适用场景

适用于壳中灯 Core 系统、Wiki、应急指南、物资库存、日记记录、成员信息和离线资料包的备份管理。

这篇内容用于防止资料只存在一台设备上，一坏就像书库着火。

## 准备材料

### 备份对象

优先备份：

- Wiki 原稿
- PocketBase 数据
- 应急指南库
- 物资库存
- 日记记录
- 成员资料
- 系统配置
- 本地模型配置
- 重要离线文档
- 导入脚本和项目代码

### 备份介质

可使用：

- 外置 SSD
- U 盘
- 机械硬盘
- SD 卡
- 光盘
- 纸质打印件
- 冷备份设备

不要把所有备份都放在同一个包里。

## 操作步骤

### 核心原则

1. 重要资料至少保留多个副本。
2. 副本应分布在不同介质上。
3. 至少保留一个离线冷备份。
4. 备份后要能恢复，不能只复制。
5. 备份要有日期、版本和校验记录。

### 1. 建立备份目录

建议格式：

```text
backup/
  2026-06-16/
    pocketbase/
    wiki_import/
    emergency/
    app_config/
    notes/
```

### 2. 执行复制

复制时优先使用完整目录结构，不要只复制零散文件。

### 3. 记录版本

记录：

- 备份日期
- 备份人
- 来源设备
- 目标介质
- 备份内容
- 是否校验
- 是否测试恢复

### 4. 做恢复测试

至少抽查：

- Markdown 能否打开
- PocketBase 数据是否能恢复
- JSON 文件是否能读取
- 页面是否能访问
- 搜索是否正常

### 备份频率

建议：

- 每次重大修改后备份一次
- 每周做一次普通备份
- 每月做一次冷备份
- 每季度检查冷备份可读性

## 判断标准

### 检查点

定期检查：

- 是否有至少两个副本
- 是否有离线副本
- 是否有备份日期
- 是否能恢复
- 介质是否损坏
- 是否缺少最新 Wiki
- 是否缺少数据库文件
- 是否只备份了页面没备份数据

## 风险提示

### 常见错误

- 只备份代码，不备份数据库。
- 只备份数据库，不备份 Markdown 原稿。
- 所有备份放在同一块硬盘。
- 从未测试恢复。
- 文件名没有日期。
- 覆盖旧备份后发现新备份损坏。

### 停止条件

出现以下情况时，应暂停继续覆盖旧备份：

- 源文件疑似损坏
- 数据库异常
- 目标硬盘报错
- 复制过程中断
- 无法确认备份完整
- 旧备份是唯一可用副本

## 记录建议

### 备注

壳中灯系统建议保留：

- 热数据：当前运行目录
- 温备份：常用外置 SSD
- 冷备份：断电保存设备或只读介质
- 纸质摘要：关键说明和恢复步骤

备份的价值不是“有文件”，而是“能把系统救回来”。
